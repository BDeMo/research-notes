\begin{frame}{Background: About RCA}
  \begin{itemize}
    \item \textbf{Core need}: turn noisy operational evidence into an actionable
      root-cause diagnosis and recovery recommendation.
    \item RCA outputs are structured: failing service, fault type, evidence,
      confidence, severity, and recommended fix.
    \item Inputs are evidence-heavy: symptoms, service topology, logs, traces,
      fault reports, historical cases, and human summaries.
    \item A useful RCA model must connect evidence to cause, not only generate
      fluent incident text.
  \end{itemize}
\end{frame}

\begin{frame}{Significance: Why an RCA Foundation Model}
  \begin{itemize}
    \item General LLMs have strong language ability, but they are not specialized
      for structured RCA outputs or service-level diagnosis.
    \item An RCA foundation model should provide a reusable base capability for:
      \begin{itemize}
        \item root-cause identification;
        \item evidence extraction and grounding;
        \item fault / severity / confidence classification;
        \item fix recommendation generation.
      \end{itemize}
    \item The goal is not only a demo UI; the goal is to train a model with
      reusable RCA capability.
  \end{itemize}
\end{frame}

\begin{frame}{Main Goal: Train an RCA Foundation Model}
  \begin{itemize}
    \item Start from existing open-source pretrained models.
    \item Train them on RCA-style data so they learn domain behavior that the
      original pretrained model does not reliably have.
    \item Use public RCA datasets as the core training/evaluation source:
      \begin{itemize}
        \item Nezha;
        \item OpenRCA-500;
        \item RCAEval;
        \item lincyaw/rca.
      \end{itemize}
    \item Use a small internal RCA dataset only as an internal qualitative probe,
      not as a public release artifact.
  \end{itemize}
\end{frame}

\begin{frame}{Deliverables: What We Have Built}
  \begin{itemize}
    \item \textbf{Main deliverable}: an RCA-capable foundation-model checkpoint
      trained from open-source base models on public RCA datasets.
    \item The model can be used as an internal RCA probe:
      \begin{itemize}
        \item load checkpoint in `rca-chat`;
        \item run public RCA cases for reproducible evaluation;
        \item run private RCA cases only for internal qualitative inspection.
      \end{itemize}
    \item Supporting deliverables:
      \begin{itemize}
        \item RCA SFT matrix over Qwen2.5 / Qwen3, curriculum / joint-mix,
          4 public RCA datasets, and 5 general benchmarks;
        \item matrix reports and provenance notes;
        \item project brief slides for stakeholder communication.
      \end{itemize}
  \end{itemize}
\end{frame}

\begin{frame}{Training Design: Curriculum vs. Mix}
  \begin{columns}[T]
    \begin{column}{0.48\textwidth}
      \begin{block}{Curriculum Line}
        \begin{itemize}
          \item Train sequentially by dataset order:
            Nezha $\rightarrow$ OpenRCA-500 $\rightarrow$ RCAEval $\rightarrow$ lincyaw.
          \item Each later step starts from the previous trained checkpoint.
          \item Goal: test whether RCA capability can accumulate across datasets.
          \item Key property: exposes retention / forgetting after each new RCA domain.
        \end{itemize}
      \end{block}
    \end{column}
    \begin{column}{0.48\textwidth}
      \begin{block}{Mix Line}
        \begin{itemize}
          \item Combine all public RCA datasets into one shuffled training set.
          \item Train one model with balanced exposure to all RCA domains.
          \item Goal: get the strongest single RCA checkpoint across datasets.
          \item Key property: stronger average RCA behavior, but can still affect general ability.
        \end{itemize}
      \end{block}
    \end{column}
  \end{columns}

  \vspace{0.4em}
  \begin{itemize}
    \item This two-line design lets us separate two questions: can RCA skills
      accumulate over time, and what is the best single checkpoint for a demo?
  \end{itemize}
\end{frame}

\begin{frame}{Future Problem: Catastrophic Forgetting}
  \begin{itemize}
    \item The largest research issue is \textbf{catastrophic forgetting}.
    \item Definition: after training a pretrained model on new RCA data, the model
      may lose capability learned from earlier RCA datasets or from pretraining.
    \item This happens because our RCA data is much smaller than the original
      pretraining corpus; we cannot fully protect the base model by simply
      mixing in all original data.
    \item \textbf{Observed in v2}: lincyaw `service\_set\_f1` dropped
      0.563 $\rightarrow$ 0.233 after the anti-forget quick-win retrain.
    \item \textbf{Observed in v2}: Nezha `evidence\_overlap` dropped
      0.578 $\rightarrow$ 0.000 even though JSON parseability improved.
    \item \textbf{Observed on q3}: cumulative checkpoints showed format collapse,
      with RCAEval `json\_parseable` down to 0.113 at $q3_{curr\_4}$.
    \item \textbf{Observed on general benchmarks}: $q25_{mix}$ dropped GSM8K
      0.795 $\rightarrow$ 0.715 and TruthfulQA 0.647 $\rightarrow$ 0.589.
  \end{itemize}
\end{frame}

\begin{frame}{Future Work: How We Are Trying to Solve It}
  \begin{itemize}
    \item We already compared multiple training strategies:
      \begin{itemize}
        \item sequential curriculum training;
        \item joint-mix training across RCA datasets;
        \item anti-forget v2 with data balancing and a small general anchor.
      \end{itemize}
    \item Early conclusion: direct SFT is useful for initializing RCA capability,
      but it is not stable enough as the final answer.
    \item Next research focus: keep the base model more stable and move
      adaptation into memory / wrapper mechanisms.
    \item Next demo: compare full context, summary, retrieval, and
      compressed-memory RCA on the same public RCA case.
  \end{itemize}
\end{frame}
