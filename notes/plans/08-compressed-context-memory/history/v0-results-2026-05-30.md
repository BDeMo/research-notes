## TL;DR

1. **The original `coding_niah` task is not solvable by any tested
   wrapper.** Baselines `full_context` and `retrieval` reach
   `contains=1.000` after fixes; all four wrappers (`prefix`,
   `residual`, `interleave`, `hybrid`) stay at `0.000` on the
   held-out split even when they reach `0.99` train CE. This is an
   architectural ceiling, not a bug.
2. **Switching to the 2-bit `categorical_niah` task validates the
   architecture.** With 2000 training steps, `interleave` hits
   `train_contains=0.920` (proving the recurrence can learn the
   chunk→class mapping), and the new `xattn` combine reaches
   `eval_contains=0.420` — well above the `summary` baseline
   (`0.100`), but still below `full_context` (`0.720`) and
   `retrieval` (`0.900`).
3. **`xattn` (apply-time chunk-pool readout) is the best wrapper.**
   It is the only mode with: (a) the highest eval accuracy across
   the matrix, (b) a *non-collapsed* prediction distribution
   (balanced across all 4 classes), and (c) a manageable
   train→eval gap (+0.24, vs `interleave`'s +0.64).
4. **The dominant failure mode is *collapse*, not
   *under-fitting*.** Three out of five combines collapsed to a
   single-class default on `categorical_niah` (e.g. `hybrid` says
   "concat" for 48/50 eval items). The collapse is what kills
   `contains` even when CE goes down — the wrapper learns a constant
   prefix that biases the model toward one class.
5. **Two non-architectural fixes mattered more than expected.**
   (a) Left-truncation in `tokenize_to_ids` (the question was being
   cut off by right-truncation, sinking baseline `contains` from
   `1.000` to `0.000`). (b) Stop-on-newline in `generate_greedy`
   (verbose continuations after the answer were depressing
   `token_f1` from ~0.5 to ~0.07 even when `contains_match` was
   `1.0`).

## Datasets

### `coding_niah` (40-bit answer)

Synthetic 8-chunk Python "files" with one needle chunk containing

```python
def foo_<6 lowercase hex>():
    """Returns the secret token planted in this synthetic case."""
    return "<8-char uppercase alphanumeric>"
```

The gold answer is the 8-char string. Answer entropy ≈ 40 bits
(`base32^8`).

### `categorical_niah` (2-bit answer)

Same haystack scaffolding, but the planted function computes one
of four operations:

```python
def foo_<id>(a, b):
    """Computes the {op} planted in this synthetic case."""
    return a + b       # gold = "sum"
    # or  a * b        # gold = "product"
    # or  max(a, b)    # gold = "max"
    # or  str(a)+str(b)# gold = "concat"
```

Query: `"Which operation does the function foo_<id> compute?
Respond with exactly one of these labels and nothing else:
sum, product, max, concat."` Answer entropy = 2 bits.

The two datasets share generator + chunk distribution → the only
difference is the answer-space cardinality. This makes the
"architecture can't recall 40-bit strings but can recall 2-bit
classes" claim a clean apples-to-apples comparison.

## Combine modes (the "how is memory delivered to the base" axis)

All combine modes share the per-step recurrence

```
candidate = m_{t-1}
for block in update_blocks:
    candidate = block(candidate, chunk_hidden, chunk_mask)  # memory Q × chunk KV
delta = candidate - m_{t-1}
m_t = m_{t-1} + α · delta
```

and differ only in what they do at *apply* time:

| Mode | Delivery at apply time | Extra trainable block |
|---|---|---|
| `prefix` | `inputs_embeds = [m, x]` | — |
| `interleave` | `inputs_embeds = [x[:p], m, x[p:]]`, `p = ⌊T·0.5⌋` | — |
| `residual` | `e_t' = e_t + α · CrossAttn(e_t, m)` per base-token | per-token cross-attn |
| `hybrid` | `residual` then `prefix` | per-token cross-attn |
| **`xattn`** *(new)* | `attended = CrossAttn(m, concat(chunks_hidden))`, then `inputs_embeds = [attended, x]` | apply-time cross-attn over the **raw chunk pool** |

`xattn` was added in this round of work. The motivation:
recurrent compression has to throw away chunk-level token detail
to fit `K=32` memory slots, but at apply time we can let the
post-recurrence memory query the *full* chunk pool one more time
with its own learnable Q/K/V/LN/α. The wrapper's `update()` now
appends the projected chunk hidden states to
`memory.extra["chunks_hidden"]`; `apply()` concatenates them and
runs the readout block.

## Results

### Iteration 1 — `coding_niah` (40-bit answer)

Run: `train-20260530-101226UTC-real-1k` (100 train items,
500 steps, K=32, single split with train==eval to measure
memorization).

| Strategy | `contains` | `token_f1` | tokens |
|---|---:|---:|---:|
| `full_context` (baseline) | **1.000** | 0.072 | 1 433 |
| `retrieval` (baseline) | **1.000** | 0.074 | 520 |
| `summary` (baseline) | 0.100 | 0.014 | 173 |
| `prefix` (wrapper) | 0.000 | 0.000 | 70 |
| `residual` (wrapper) | 0.000 | 0.000 | 38 |
| `interleave` (wrapper) | 0.000 | 0.000 | 70 |
| `hybrid` (wrapper) | 0.000 | 0.000 | 70 |

Training CE on coding_niah dropped from ~3.0 to ~0.99 over the
500 steps for `prefix`/`interleave` (so the wrapper *did fit* in
loss terms), but the actual `contains` stayed flat at 0. The
model learns the conditional distribution of *the first answer
token under teacher forcing* but never produces the right token
under free-form generation. This is consistent with a
soft-prompt architecture that biases the base's logits but does
not carry enough exact-string information through `K=32` tokens.

### Iteration 2 — `categorical_niah` (2-bit answer), 2000 steps

Run: `train-20260530-215611UTC-cat-long` (100 train + 50
held-out eval, 2000 steps, K=32).

Baselines:

| Strategy | `eval_contains` | `eval_f1` | tokens |
|---|---:|---:|---:|
| `retrieval` | **0.900** | 0.303 | 528 |
| `full_context` | 0.720 | 0.245 | 1 429 |
| `summary` | 0.100 | 0.026 | 172 |

Wrappers (now with train-eval split):

| Combine | `eval_c` | `eval_f1` | `train_c` | `train_f1` | gap | first-token dist (eval) |
|---|---:|---:|---:|---:|---:|---|
| **`xattn`** | **0.420** | 0.211 | 0.660 | 0.404 | +0.24 | sum 14 / product 14 / max 2 / concat 20 |
| `prefix` | 0.300 | 0.237 | 0.480 | 0.400 | +0.18 | NONE 18 / product 17 / concat 11 / max 4 |
| `interleave` | 0.280 | 0.202 | **0.920** | 0.693 | **+0.64** | sum 17 / product 18 / concat 12 / max 3 |
| `hybrid` | 0.200 | 0.200 | 0.300 | 0.280 | +0.10 | concat 48 / sum 2 (collapsed) |
| `residual` | 0.000 | 0.000 | 0.020 | 0.004 | +0.02 | NONE 49 / product 1 (collapsed) |

Observations:

1. **`xattn` wins on eval and stays well-distributed.** It is the
   only wrapper that produces all four class labels in non-trivial
   counts and avoids the "default to one class" failure mode.
2. **`interleave` shows the architecture *can* learn the task** —
   `train_contains=0.920` is far above chance (`0.25`) and is
   close to `full_context`. The 64-point train→eval drop is a
   classic small-data overfit.
3. **`hybrid` and `residual` collapse to a constant prediction.**
   Their CE goes down in training but the actual generation
   collapses onto one class. `residual`'s `α_app` barely moves
   from its `0.0` init (only `0.009` after training), so the
   per-token residual channel never engages.
4. **The summary baseline is the floor (0.10) and `retrieval` is
   the ceiling (0.90).** `xattn`'s `0.42` is roughly the midpoint;
   `prefix`'s `0.30` is a third of the way up.

### Iteration 0.5 — what didn't work and how we knew

Two prompt/decoding bugs almost killed the baselines:

| Bug | Symptom | Fix |
|---|---|---|
| Right-truncation in `tokenize_to_ids` | `full_context`/`retrieval` baselines stuck at `contains=0` even on toy examples | Switch to **left-truncation** so the `### Question … ### Answer` suffix is preserved |
| No stop criterion on `generate_greedy` | Model outputs correct answer + 40 tokens of "Okay, let me think…" → `contains=1.0` but `token_f1≈0.07` | Default `stop_strings=("\n",)`, drop the matched suffix |

Both are now in `llm_infra` and apply uniformly to all baselines and
wrappers.

## Diagnostic geometry

The full geometry panel is in `summary.json` (`drift_curve`,
`delta_norms`, `consecutive_delta_cosine`, `delta_chunk_cosine`).
Concrete readings on the first eval item from the 2000-step
`categorical` run (each list spans 5 consecutive Δm transitions
across the 6 chunks):

| Combine | `cos(Δm_t, Δm_{t-1})` | `cos(Δm_t, c_t)` | `‖Δm‖` | drift_final |
|---|---|---|---|---|
| `hybrid`     | `[1.00, 1.00, 1.00, 1.00, 1.00]` | `≈ +0.02` | `~3 240` | 19 328 |
| `prefix`     | `[0.57, 0.92, 0.98, 0.98, 0.99]` | `≈ +0.005` | `~2 010` | 11 200 |
| `xattn`      | `[-0.88, 0.93, 0.99, 0.99, 1.00]` | `≈ +0.008` | `~1 650` | 7 648 |
| `interleave` | `[-0.52, 0.92, 0.90, 0.83, 0.86]` | `≈ -0.002` | `~1 450` | 5 568 |
| `residual`   | `[0.39, 0.41, 0.82, 0.75, 0.84]` | `≈ +0.004` | `~2 270` | 10 240 |

Three things stand out:

1. **`hybrid`'s consecutive-Δ cosine is exactly `1.000`.** Every
   chunk produces the same update direction. This is a fully
   degenerate recurrence — collapse is built into the geometry,
   not just the output distribution.
2. **`xattn` and `interleave` start with a sharply *anti-aligned*
   first transition (`cos ≈ -0.88` / `-0.52`)** and then converge
   to the high-cosine plateau. The first chunk's update *is*
   informative; after that the recurrence runs out of new
   directions to move in.
3. **`cos(Δm_t, c_t) ≈ 0` for every combine.** The Δm direction
   is essentially uncorrelated with the chunk content direction.
   This is partly a high-dim artifact (`K·d ≈ 131k`, where random
   pairs have cosine `O(d^{-1/2})`), but combined with the
   high consecutive cosine it tells a clear story: the update is
   approximately a *fixed direction in memory space, repeated
   each chunk*. That direction nudges the base toward one
   answer-class regardless of input.

The `lambda_div=0.1` regularizer pushes consecutive cosine *down*
slightly but is not strong enough to break the plateau (compare
the `divhi05` variant currently in flight, which sets `λ_div=0.5`).
This is the most actionable lever in the next round.

## `xattn` variant sweep

Four `xattn` variants run in parallel on `sam-dev`, one per GPU
(`runs/explore-20260530-231621UTC-xattn/`). All share
categorical_niah, K=32, 2000 steps, max_new=16, grad_clip=1.0,
λ_KL=1.0; differences and results:

| Variant | Diff vs reference xattn | `eval_c` | `eval_f1` | `train_c` | `train_f1` | `‖Δm‖_p50` |
|---|---|---:|---:|---:|---:|---:|
| **reference** | (none) | **0.420** | 0.211 | 0.660 | 0.404 | 1 650 |
| `data300` | `--n-items 300` (3× train items, same 2000 steps → ⅓ epochs) | 0.340 | 0.071 | 0.280 | 0.058 | **8 800** |
| `dropout15` | `--dropout 0.15` (attention + FFN dropout) | 0.280 | 0.151 | **0.840** | 0.531 | 802 |
| `divhi05` | `--lambda-div 0.5` (5× Δm-diversity reg) | 0.100 | 0.068 | 0.480 | 0.223 | 5 856 |
| `lrlo2e5` | `--lr 2e-5` (vs `5e-5`) | 0.260 | 0.227 | 0.800 | 0.800 | 394 |

Headline finding: **none of the four perturbations beat the
reference recipe on eval `contains`.** Three of the four
*increase* the train-eval gap (i.e. they make the wrapper overfit
*more*, not less). One observation per variant:

- `data300` is **under-trained**, not overfit-limited. 2000 steps
  spread over 300 items is only ~6.7 epochs vs the reference's 20.
  Both train and eval `contains` *drop* and `‖Δm‖_p50` jumps to
  8 800 — the wrapper is taking huge, unstable update steps and
  has not yet settled into either a memorization or a generalizing
  regime. The natural follow-up is `data300` with 6 000 steps
  (matched epochs).
- `dropout15` hurts eval (-0.14) but *raises* train to `0.840` (the
  highest of any wrapper run in this report). Attention dropout
  during training does not act as the generalization regularizer
  we hoped; it appears to push the wrapper toward stronger
  per-item memorization while randomizing the chunk-pool
  attention at eval time.
- `divhi05` collapses eval to `0.100`. The strong Δm-diversity
  penalty forces the recurrence to take *different* directions
  each chunk, but those directions are uninformative — the
  wrapper basically reverts to a noisy version of the default
  prefix. This rules out "the consecutive-cosine plateau is the
  problem" as a working hypothesis.
- `lrlo2e5` finds an *even tighter* memorization solution
  (`train_c=train_f1=0.800`) at lower eval. Smaller steps converge
  to a sharper minimum that fits the training items more
  precisely and generalizes less. `‖Δm‖_p50` is the smallest of
  any wrapper run (394) — tiny, precise per-chunk nudges that the
  base model has memorized but cannot extrapolate.

So the variant sweep tells us: **the reference recipe is
already at a local optimum of this hyperparameter manifold.**
The next round needs either (a) more training tokens with
matched epochs, (b) architectural change (true KV injection
into the base, real cross-attention adapter), or (c) acceptance
that K=32 soft memory has fundamental capacity limits on this
task family.

## Verdict so far

Two pieces of v0's original claim are validated, one is contradicted.

| v0 claim | Status | Evidence |
|---|---|---|
| Compact memory state `m_t` can be learned via `G_phi(m_{t-1}, c_t)` | **Validated** (on 2-bit task) | `interleave` reaches `train_c=0.92` on `categorical_niah`; `xattn` reaches `eval_c=0.42` |
| Reduces context/KV cost compared to baselines | **Validated** | Wrappers use 70 input tokens vs `retrieval`'s 528 vs `full_context`'s 1429 |
| Preserves information needed by downstream answers | **Partially contradicted** | On 40-bit `coding_niah`, all combines fail (0.00 vs 1.00). On 2-bit `categorical_niah` the best wrapper (xattn) is at 0.42 vs retrieval's 0.90 — a real gap |

The right way to frame this is: **`K=32` soft-prompt memory has
information capacity well below an explicit retrieval baseline,
and the gap shows up as answer-entropy hits a few-bits ceiling.**
The architecture works qualitatively but not at parity with
explicit retrieval, on these synthetic tasks.

## Open questions (priority order, post-variant-sweep)

1. **Matched-epochs data300 rerun (cheap, high info).** 300 items
   × ~20 epochs = ~6 000 steps. This is the single experiment that
   distinguishes "K=32 capacity is the limit" from "100 items
   isn't enough data". Fastest way to find out.
2. **True KV injection into the base model.** `xattn` currently
   only attaches the attended memory as a single prefix; the
   base's Q never sees memory beyond layer 0's input. A
   `combine=layer_kv_inject` mode would hook each
   `Qwen3Attention.forward`, project the post-recurrence memory
   through per-layer K/V adapters, and concatenate the resulting
   `(K_mem, V_mem)` to the layer's self-attention KV before
   softmax. This is the Memorizing-Transformers / Compressive-
   Transformer construction. Higher implementation cost
   (~2–3 hours) but the only intervention the variant sweep
   *didn't* test.
3. **K-sweep on `xattn`.** All current runs use `K=32`. A
   `{K=8, 16, 32, 64, 128}` sweep on the reference recipe would
   directly characterize the soft-memory capacity curve. We can
   plot `eval_c(K)` and see whether the curve saturates or keeps
   climbing.
4. **Switch from synthetic NIAH to real long-context (RepoBench-C
   / LongBench).** The synthetic variants are deliberately
   adversarial (random 8-char strings, exhaustively-labeled
   4-class). Real tasks have soft answer distributions where
   soft-prompt memory might be a better fit than for exact-string
   recall. Required before any external claim about wrapper utility.
5. **Hybrid wrapper + retrieval.** Even if `xattn` plateaus below
   `retrieval`, the two could compose: prefix the top-1 retrieved
   chunk *and* the wrapper memory. Could give the best of both
   (low-token compressed context + grounded recall).

## Reproducing the matrix

```bash
# 0. Bring up sam-dev, set up env (one-time)
bash mem-embedding/k8s/setup-env-on-pod.sh

# 1. Push current workspace to the pod
bash mem-embedding/k8s/sync-to-sam-dev.sh

# 2. Run the full 5-combine sweep with train-eval diagnostics
kubectl exec sam-dev -n <ns> -c ssh -- su - devuser -c '
cd ~/workspace/mem-test
DATASET=categorical_niah N_ITEMS=100 N_EVAL_ITEMS=50 N_CHUNKS=6 \
N_MEMORY=32 TRAIN_STEPS=2000 LR=5e-5 LAMBDA_KL=1.0 LAMBDA_DIV=0.1 \
GRAD_CLIP=1.0 MAX_NEW=16 \
COMBINES=prefix,residual,interleave,hybrid,xattn \
OUT_DIR=runs/cat-5combines GPUS=0,1,2,3 \

'

kubectl exec sam-dev -n <ns> -c ssh -- su - devuser -c '
cd ~/workspace/mem-test

'

# 4. Pull artifacts back
RUN_GLOB='*cat-5combines*' bash mem-embedding/k8s/pull-runs.sh

```
