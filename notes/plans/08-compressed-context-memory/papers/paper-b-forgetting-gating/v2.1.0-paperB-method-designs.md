# Paper B — method design space (signals → a pluggable, transfer-based long-context compressor)

> Directive 2026-07-06: **drop the gate (C3) from Paper B.** Use the dive-in **signals** to design a NEW method that
> improves long-context accuracy at *finding/keeping the important information*. Hard constraints:
> **(1)** base model unchanged, or a **pluggable module** insertable anytime; **(2)** compute- & data-limited → **transfer
> knowledge from existing base models** rather than train from scratch; **(3)** works for **both linear (SSM) & quadratic
> (attention)** AR; **(4)** **solve the long-prefill** cost; **(5)** deliverable = a **fast-reproducible checkpoint + method**;
> selling points = **general · efficient · simple · insightful**.

## 0. What the facts force the design to be (grounded in `matrix-facts.md`)
- **No single signal/method is universal** (F3 task-flip, F4 kvzip retrieval-only, F1 attention-collapse-at-16k) ⇒ the method must **combine signals** and be **content/query-adaptive**, not a fixed compressor.
- **Naive merging destroys needles** (F14) and **attention scores are unreliable at length** (F1) ⇒ importance must come from **length-robust, attention-free signals** (key-norm F2, reconstructability F4/kvzip) **+ query-relevance**, not from the base's long-range attention.
- **Full context can hurt / distractors hurt readers** (F6/F7) ⇒ **pruning is not just for speed — it can raise accuracy** (denoising). This is the performance lever.
- **RAG fails abstractive** because it uses *lexical* overlap (F9) ⇒ use the **base's own semantic signal**, not an external lexical retriever.
- **Linear/GDN has no KV lever + state-collapse** (F10, same in Qwen3.6) ⇒ linear models need an **external exact-recall side-channel for the few important tokens**.
- **Failures are architecture-level, size/family-invariant** (F11) ⇒ a signal-based module should **transfer across bases**.

**⇒ The shape of the method:** a **tiny, pluggable "importance router"** that runs a *cheap* pass over the long context (no full attention), scores each token/chunk by a **learned combination of length-robust signals + query-relevance**, then **keeps important tokens verbatim / merges redundant ones**, feeding a **short** sequence to the frozen base. The router is **distilled from the base's own behavior on a little data** (knowledge transfer, not retraining) and shipped as a **small adapter checkpoint**.

## 2. Recommended method (unifying): **"IMP" — an Importance-router prefilter, distilled & pluggable**
**Pipeline (inference):** long ctx → (a) **cheap chunk/token scoring** with attention-free length-robust signals {key-norm, reconstructability, query-dot} → (b) **keep top-p important tokens verbatim + merge redundant spans** (reconstruction-guided, D3) → (c) frozen base reads the short sequence (linear or quadratic) → (d) *[linear only]* inject a small **verbatim side-cache** of the very-top tokens (D4) to beat state-collapse.

**How it satisfies every constraint:**
- **Base unchanged / pluggable:** IMP is a pre-base module + optional prefix; the base weights are frozen.
- **Transfer, not retrain (compute/data-limited):** the router is **self-distilled** from the base's *own* full-attention/retrieval-head importance on a **small** unlabeled corpus (a few thousand long-doc examples). No task labels, no base retrain. This is the **weight-transfer**: the base already encodes "what's important"; IMP extracts it into a cheap scorer.
- **Cross-base / general (F11):** because failures are architecture-level, IMP transfers — train the tiny router once per base (minutes), or **warm-start from another base's router** (the importance notion is largely base-agnostic; only a light re-fit needed).
- **Solves long-prefill:** scoring is O(L) (embeddings/norms, no L² attention); the expensive base read only sees the kept short sequence.
- **Efficient · simple · insightful:** one small module; grounded in exactly why each baseline fails (§0); the "keep-what's-unreconstructable, drop-what's-redundant, route-by-query" rule is a one-sentence method.
- **Fast-reproducible checkpoint:** ship **base + IMP-adapter** (small); "reproduce" = load base + the tiny adapter, or run the minutes-long self-distill. If we release the adapter, others get a long-context-capable checkpoint instantly.
- **Linear + quadratic:** the prefilter is architecture-agnostic; the side-cache (D4) is the linear-only add-on for exact recall.

**Novel insights (beyond the read papers) it exploits:**
1. **Ensemble-of-signals, task-adaptive** — since no single signal wins (F3/F4), *learn* to combine them; the router's win over any single-signal baseline is the empirical contribution.
2. **The base is its own importance oracle** — self-distill retrieval-head/attention importance into a cheap scorer (no external retriever, no labels) → dodges RAG's lexical failure (F9) and prompt-compression's needle-drop (F8).
3. **Pruning as accuracy (not just speed)** — F6/F7 say removing distractors can *raise* accuracy; IMP is positioned as a **performance** method, not an efficiency one.
4. **Un-reconstructability = keep-verbatim** — merge the reconstructable (redundant) and protect the un-reconstructable (F4 inverted): the needle is exactly the token you *can't* reconstruct from its neighbors.

## 3. Extension dimensions (where we can grow it)
- **Signal set:** add/ablate {key-norm, reconstructability, query-dot, retrieval-head-predicted, surprisal, redundancy} → which subset the router needs (dive-in X-C8/X-C2).
- **Granularity:** token vs chunk vs sentence scoring (chunk = cheaper prefill).
- **Keep-vs-merge policy:** hard-keep top-p, merge the rest, or soft (weighted).
- **Transfer axis:** train-per-base vs warm-start-across-bases vs a single base-agnostic router (measure cross-base transfer — ties to F11 and the "fast checkpoint" claim).
- **Architecture axis:** quadratic (short prefill saves O(L²)) vs linear (short prefill + side-cache for exact recall).
- **Budget axis:** the keep-ratio as a single knob → an accuracy/latency curve (the efficient-and-simple selling point).
- **Data axis:** how little distill data suffices (the data-limited selling point) — a data-scaling ablation.

## 4. Build order (compute-frugal)
1. **Signal harvest + MINE (X-C2/X-C8)** — measure which signals identify answer tokens & predict failure; pick the 2–3 that matter.
2. **Training-free IMP** (D1/D3 with the picked signals) — no distill yet; establish it beats ToMe/KV-evict/RAG on the necessity + task-flip benches.
3. **Distilled router (D2)** — tiny scorer self-distilled from the base's attention on a small corpus; show it beats the training-free heuristic and transfers across bases (the checkpoint).
4. **Linear side-cache (D4)** on GDN (Qwen3.5/3.6) — close the state-collapse gap.
5. **Package** base + IMP-adapter as the fast-reproducible checkpoint + a one-command repro.
