## 0. The two papers (refocused)
- **Paper A — the do-no-harm GATE (practical & general).** *Question:* how to make the gate real — **base-model intrinsic signal vs a trained gate?** *Emphasis:* **robustness** + **gating performance**, and it must **adapt to as many compression methods as possible** (KV-evict, prompt-compress, merge, retrieval — one gate over all). Deliverable: a compressor-agnostic, out-of-sample-verified gate that turns *any* lossy compressor into a "never-worse-than-feasible-baseline" system.
- **Paper B — identify & keep the important information (impactful).** *Question:* how to raise long-context **accuracy at finding the important info**, and how to **identify** what's important. *Emphasis:* **performance**, built on a large **insight/dive-in** program; then design a **general** method that lifts many models. Method path starts simple (**pruning → token-merging**) and grows toward **importance-guided** keeping.

## 1. Paper B method roadmap (merging-first)
1. **ToMe (token merging), training-free** — *implemented* (`GCM_BASELINE=tome`, `run_baseline.py`). Input-side bipartite soft-matching (Bolya 2022, [ToMe](https://arxiv.org/abs/2210.09461)): iteratively merge the most-similar ctx token pairs (cosine on embeddings), weighted by merge "size", **order-preserving** for the causal LM. Knobs: keep-ratio, sim space (embed / random-control). *Early result:* at 50% keep it **destroys retrieval** (RULER 0.0) and hurts squad (0.23 < no-ctx) — naive averaging loses the needle ⇒ **key-knowledge loss** (the R-MeeTo diagnosis). This *motivates* the next steps.
2. **Importance-guided merging** — instead of merging by raw similarity, **protect high-importance tokens** (from the dive-in metrics §3: MI-to-answer, retrieval-head attention, key-norm, reconstructability) and only merge low-importance/redundant spans. Hypothesis: keeps the needle while shrinking filler ⇒ beats both ToMe and KV-evict in the necessity regime.
3. **SSM/linear redesign (§4)** — on a GDN base, merging + **brief re-training** (R-MeeTo) to rebuild the state's "key knowledge."

## 3. Dive-in observation metrics (≥10; attention + information-theoretic + MINE)
> Setup: fix a model + a long-ctx bench with a known answer span; collect per-token / per-layer tensors; correlate each metric with **per-item correctness** and with **being the answer-bearing token**. Goal: metrics that (a) *identify* important tokens and (b) *predict* where compression/merging will fail. 🟢 cheap · 🟡 medium · 🔴 heavy.

**Attention / circuit signatures**
1. 🟢 **Answer-position → context attention map** — where the last query token attends across ctx positions (per layer/head). The direct "what does the model look at."
2. 🟢 **Prefix / attention-sink mass** — fraction of attention on the first few tokens vs the rest ([sinks](https://arxiv.org/abs/2309.17453)); does sink mass shift with length/task?
3. 🟢 **Attention entropy by layer** — dispersion; attention sparsifies with depth ([PyramidKV](https://arxiv.org/abs/2406.02069)).
4. 🟡 **Retrieval-head engagement** ([Wu 2025](https://arxiv.org/abs/2404.15574)) — activation of known retrieval heads on the needle; ablate → NIAH breaks.
5. 🟡 **Attention-rollout / gradient-saliency to the answer logit** — causal per-token importance for the correct answer.

**Information-theoretic (MINE-based — [Belghazi 2018](https://arxiv.org/abs/1801.04062))**
6. 🔴 **I(token-hidden ; answer)** via MINE — estimate mutual information between each ctx token's representation and the gold answer → an **info-theoretic token-importance score** (the principled "which tokens carry the answer").
7. 🔴 **I(compressed/merged memory ; full-ctx hidden)** via MINE — how much information the compressed representation retains about the original context (quantifies lossiness per method/ratio).
8. 🟡 **I(token ; query)** via MINE — query-relevance of each token (drives query-aware keeping).
9. 🟢 **Token self-information / surprisal** (LM perplexity) — the LLMLingua/Selective-Context signal; low-info tokens are safe to drop.

**Norm / reconstruction / representation**
10. 🟢 **Key-L2-norm distribution** (the Knorm signal) — correlate key-norm with answer-token-ness.
11. 🟢 **Per-token reconstructability** (the KVzip signal) — can the token be recovered from the compressed state? recoverability = importance.
12. 🟡 **Logit-lens answer-decodability by layer** — at which depth does [ctx] make the answer decodable (where info consolidates).
13. 🟢 **CKA(merged-ctx hidden, full-ctx hidden)** — representational fidelity lost to merging/compression.
14. 🟢 **Effective rank / participation ratio of the ctx representation** — intrinsic dimensionality ⇒ how compressible this input is.

**Behavioral**
15. 🟢 **Needle-depth recovery curve** ("lost-in-the-middle") — accuracy vs needle position; interacts with merging.
16. 🟢 **Adjacent-token key redundancy** (cosine of neighbor keys) — the merge-ability signal that ToMe exploits; high redundancy = safe to merge.

**Selection rule (same as before):** keep metrics that both **separate answer vs filler tokens** (large effect size) **and predict** per-item correctness under compression. Likely survivors to build the method on: **#6 (MI-to-answer), #4 (retrieval-head), #10 (key-norm), #11 (reconstructability), #16 (redundancy)** — these give a computable "protect this token" score for importance-guided merging.

**MINE experiment plan:** train a small MINE critic on (token-hidden, answer-embedding) pairs pooled over a bench; report I estimates per token-type (answer vs filler) and per condition (in/cross-domain, per compression method/ratio). Deliverable: an info-importance ranking we can plug into the merging keep-decision, and I(M;full) curves per compressor.

## 4. SSM / linear-complexity base redesign (R-MeeTo-style)
On a **GDN base (Qwen3.5/3.6)** there's **no KV cache** — token reduction must happen on the *input/state* side, and (per [R-MeeTo](https://github.com/NUS-HPC-AI-Lab/R-MeeTo), [arXiv:2412.12496](https://arxiv.org/abs/2412.12496)) naive token reduction on Mamba/SSM causes **key-knowledge loss** far worse than on Transformers, but a **brief merge + re-training** rebuilds it (Vim-S recovered to −1.3% in minutes). Plan: (a) apply importance-guided merging to the GDN input stream; (b) **short re-training** (LoRA on the frozen GDN base) to let the recurrent state re-absorb merged-token knowledge; (c) measure whether merge+retrain closes the retrieval gap that naive merge (§1) opens. This is the linear-model arm of Paper B's general method.

## 5. Qwen3.5 vs Qwen3.6 (does 3.6 still use the linear GDN module? — **yes**)
Sources: [Qwen3.6-27B HF card](https://huggingface.co/Qwen/Qwen3.6-27B), [architecture writeup](https://ai-beat.github.io/news/2026/04/qwen36-hybrid-attention-local-coding/), [KV-cache analysis](https://wal.sh/research/qwen3.6-local-first-inference/).

| | **Qwen3.5** (Feb 2026) | **Qwen3.6** (Apr 2026) |
|---|---|---|
| linear module | **Gated DeltaNet** (linear) + Gated Attention hybrid | **same Gated DeltaNet** + Gated Attention hybrid ✅ |
| hybrid ratio | 3:1 (GDN:attention) | **3:1** — 64 layers = 16 × [3×(GDN→FFN) → 1×(GatedAttn→FFN)]; only **16/64 layers carry KV** |
| flagship size | **397B-A17B** MoE (first release) | **27B dense** + **35B-A3B** MoE (small, local-first) |
| GDN heads | (family) | 48 V heads, 16 QK heads, head-dim 128 |
| Gated-Attn heads | (family) | 24 Q, 4 KV, head-dim 256, RoPE-dim 64 |
| context | long (YaRN) | **262K native**, ~1M via YaRN |
| focus | multimodal + broad capability, RL scale | **coding / real-world utility / stability**; adds **MTP** (multi-token prediction) |
| headline | general flagship | **Qwen3.6-27B beats Qwen3.5-397B on SWE-bench Verified (77.2 vs 76.2)** at a fraction of size |

**Bottom line:** 3.6 is **architecturally continuous with 3.5** (same GDN linear + gated-attn 3:1 hybrid, KV only on 1/4 of layers) — so our GDN-base findings and the SSM redesign (§4) carry directly to 3.6. The differences are **scale (much smaller), a coding/agentic focus, an explicit head layout, MTP, and local-first KV economy** — not the linear mechanism.
