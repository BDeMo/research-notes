# Self-improvement

> Methods where a model **generates the training signal for itself** — by producing candidate outputs, having a verifier (or itself) score them, and using the scored data as the next training (or in-context) update. Covers both **prompt-side** loops (the model rewrites its own response in context) and **weight-side** loops (the scored data trains the model, or its critic).

## Structure

**Contained by**: *training paradigms* (broader; not a folder here yet).

**Contains**:
- **STaR family** — generate CoT, filter by verifier, SFT the surviving traces. [star], [vstar], [restem].
- **Self-critique / iterative refinement** — model criticizes and rewrites its own output without backprop. [self-refine], [reflexion].
- **Self-rewarding** — model emits both responses and reward judgments; closes the RLHF loop without external annotators. [self-reward].
- **Preference-style updates** — pairwise comparisons (sometimes generated by the model itself) drive policy updates. [dpo].
- **Verifier-augmented self-training** — explicit verifier (PRM, programmatic checker) gates which generations survive. Bridges into [`inference-time-compute`](../inference-time-compute/).

## Key concepts

- **Verifier vs critic** — *Verifier*: programmatic / hard signal (math answer correct, code passes tests, PRM score). *Critic*: learned soft signal (reward model, judge LLM). Self-improvement quality is dominated by verifier quality.
- **Filtering vs preference** — filtering keeps only successes (STaR); preference learns from successes-vs-failures pairs (DPO, [self-reward]). Preference is denser signal but more sensitive to reward-model bias.
- **Prompt-side vs weight-side** — [self-refine] and [reflexion] keep updates in the *prompt* (X-side). STaR family writes updates into *weights*. Plan 08 pushes the weight-side variant inside a single rollout.
- **Reward hacking** — self-improvement amplifies any bias in the reward signal. Verifier hacking is the failure mode where the model finds inputs that fool the verifier without solving the task.
- **Termination / collapse** — without diversity controls, self-training can collapse to a single high-reward mode (entropy crash).
- **Off-policy / on-policy** — STaR is off-policy (train on past generations); modern variants are increasingly on-policy.

## Open questions / live debates

- **Verifier quality is the ceiling** — most self-improvement runs are bottlenecked by reward-model error. How do we improve verifiers faster than policies?
- **Reward hacking long-term** — does the model eventually find pathological inputs that fool the verifier? Empirically yes; mitigation is open.
- **Off-policy data → entropy crash** — repeated self-training can shrink the output distribution. Sample-replay / KL anchoring help but aren't principled.
- **Self-improvement vs distillation from stronger model** — when is self-improvement actually better than distillation from GPT-4-class? Boundary unclear.
- **Weight-side self-improvement** — replacing the prompt rewrite of [reflexion] with a weight delta (plan 08) — does it work? Open question for this repo.
